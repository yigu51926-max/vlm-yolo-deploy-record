# YOLO26 与 Qwen3-VL 大小模型协同策略设计

## 一、设计目标

本系统面向多路实时视频检测任务，采用“小模型实时检测 + 大模型语义分析”的协同方式。

其中，YOLO26 作为小模型，负责对多路 RTSP 视频流进行高频目标检测，快速输出目标类别、置信度和位置坐标。

Qwen3-VL 作为大模型，负责对关键帧进行图像语义理解，分析场景内容、目标关系和潜在风险，并输出自然语言解释。

系统目标不是让大模型处理每一帧视频，而是通过 YOLO26 先筛选关键帧，再将少量关键帧交给 Qwen3-VL 分析，从而兼顾实时性和语义理解能力。

## 二、模型分工

### 1. YOLO26 的作用

YOLO26 主要负责以下任务：

- 实时读取多路 RTSP 视频流
- 对每一帧进行目标检测
- 输出目标类别、置信度和检测框位置
- 判断是否出现关键目标
- 触发关键帧保存
- 为 Qwen3-VL 提供结构化检测结果

YOLO26 的优势是速度快，适合高频运行，因此作为系统的前端检测模型。

### 2. Qwen3-VL 的作用

Qwen3-VL 主要负责以下任务：

- 对 YOLO26 触发的关键帧进行视觉语义分析
- 描述画面中的主要目标和场景内容
- 结合 YOLO 检测结果判断是否存在异常情况
- 输出报警原因、风险说明和处理建议
- 为后续后台系统生成可读的事件日志

Qwen3-VL 的优势是语义理解能力强，但推理速度较慢，因此不适合逐帧运行，只适合处理关键帧。

## 三、协同流程

系统整体协同流程如下：

1. 多路 RTSP 视频流输入系统
2. YOLO26 对每路视频进行实时目标检测
3. 系统根据检测结果判断是否触发关键事件
4. 如果满足触发条件，则保存当前帧为关键帧
5. 将关键帧和 YOLO 检测结果发送给 Qwen3-VL
6. Qwen3-VL 对关键帧进行语义分析
7. 系统保存分析结果，包括视频流编号、时间戳、目标类别、置信度和大模型分析文本
8. 后续可将结果接入后台系统进行展示和报警

## 四、关键帧触发规则

为了避免大模型处理过多无效图像，本系统采用事件触发机制。

当前阶段设计以下触发规则：

### 1. 人员触发

当 YOLO26 检测到 person 类目标，且置信度高于设定阈值时，保存当前帧并触发 Qwen3-VL 分析。

默认条件：

- 类别为 person
- 置信度大于等于 0.60

### 2. 多目标组合触发

当画面中同时出现 person 和 car、truck、bus 等目标时，认为可能存在人员与车辆共处场景，触发 Qwen3-VL 进一步分析。

### 3. 周期性抽帧触发

即使没有明显异常，也可以每隔一定时间抽取一帧送入 Qwen3-VL，作为周期性巡检。

默认周期：

- 每一路视频每 30 秒最多触发一次

### 4. 冷却时间限制

为了避免同一路视频连续触发过多大模型分析，需要设置冷却时间。

默认冷却时间：

- 同一路视频两次 Qwen3-VL 分析之间至少间隔 10 秒

## 五、事件日志设计

每次触发 Qwen3-VL 分析后，系统需要保存事件日志。

日志字段包括：

- event_id：事件编号
- stream_name：视频流名称，例如 cam1、cam2、cam3
- timestamp：事件触发时间
- keyframe_path：关键帧保存路径
- detected_objects：YOLO 检测目标列表
- trigger_reason：触发原因
- qwen_analysis：Qwen3-VL 输出的语义分析结果

## 六、当前最小原型目标

当前阶段先实现最小协同原型，不做完整后台系统。

最小原型目标如下：

1. 读取一路或多路 RTSP 视频流
2. 使用 YOLO26 进行实时检测
3. 当检测到 person 时保存关键帧
4. 调用 Qwen3-VL 分析关键帧
5. 将分析结果保存为文本日志
6. 验证 YOLO 触发 Qwen 的协同流程可行

## 七、后续扩展方向

后续可以在最小原型基础上继续扩展：

- 支持多路视频同时触发大模型分析
- 增加更复杂的事件规则
- 增加风险等级判断
- 将事件日志保存为 JSON 格式
- 接入 Web 后台展示检测结果
- 将 YOLO 检测服务和 Qwen 分析服务拆分为独立模块
- 最终使用 Docker Compose 统一部署
